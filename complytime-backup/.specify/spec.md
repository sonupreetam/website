# Specification

What the complytime.dev website delivers, describing the current implementation and planned enhancements.

## Landing Page (`/`)

The landing page is the primary entry point and communicates what ComplyTime is, what it offers, and how to get started.

### Hero Section

- **Badge**: "Open Source Compliance Automation"
- **Title**: Site title from `config/_default/hugo.toml` (rendered via `{{ .Title }}`)
- **Lead text**: From `content/_index.md` frontmatter `lead` field
- **Primary CTA**: "Get Started" linking to `/docs/getting-started/`
- **Secondary CTA**: "View on GitHub" linking to `https://github.com/complytime`

### Features Grid

Six cards under the heading "Engineering-First Compliance" with the subheading "An API-driven approach to compliance that integrates seamlessly with your development workflow."

| Card | Description |

| Intelligent Automation | Automate compliance workflows with intelligent tools that understand your infrastructure and policies. |
| Policy as Code | Define compliance requirements as code using OSCAL and integrate with your existing DevSecOps pipeline. |

Each card has an SVG icon, a title, and a description.

### Projects Section

Heading: "Our Projects" with subheading: "A suite of tools designed to streamline compliance workflows from code to audit."

Project cards are dynamically generated from `data/projects.json` (written by the Go sync tool, which derives the repo list from `peribolos.yaml` in the org's `.github` repo). Each card displays:

- **Language badge** (e.g., "Go", "Python", "Shell") with color mapping via `$langColors` dict
- **Project type** (e.g., "CLI Tool", "Automation", "Observability", "Framework", "Library") — derived from repo topics and description
- **Project name**
- **One-line description**
- **Link** to `/docs/projects/{repo}/`

New repos added to `peribolos.yaml` in the org's `.github` repo appear automatically on the next sync run — no config or template changes needed. The governance registry is the authoritative source for which repos exist.

A "View All Projects" link points to `/docs/projects/`.

## Documentation (`/docs/`)

Per-project documentation under `/docs/projects/` with auto-generated content from the Go sync tool.

- `/docs/getting-started/` — Onboarding guide (hand-maintained, committed)
- `/docs/projects/{repo}/` — Per-repo project pages (auto-generated, gitignored):
  - `_index.md` — Section index with metadata frontmatter (no body content)
  - `overview.md` — README content as a navigable child page
  - `{doc}.md` — Additional doc pages from upstream `docs/` directories (via `discovery.scan_paths`)

The sync tool auto-generates project pages from repo READMEs via the GitHub API, using `peribolos.yaml` from the org's `.github` repo as the authoritative repo list. Config-declared files in `sync-config.yaml` can sync specific docs with frontmatter injection and link rewriting. All generated content is gitignored and rebuilt from scratch on every CI run.

Each docs section has auto-generated sidebar navigation via the Doks theme's section menu partial. Repo-level sections are collapsed by default via Hugo cascade in `content/docs/projects/_index.md`.

## Projects Index (`/docs/projects/`)

A section index at `content/docs/projects/_index.md` (hand-maintained, committed) that introduces the ComplyTime project ecosystem. Uses the `{{</* project-cards */>}}` shortcode to render project cards from `data/projects.json`, grouped by type (CLI Tool, Framework, Automation, Observability, Library) with star counts.

Each project card shows:

- Language badge with color mapping
- Project type
- Star count (if > 0)
- Project name
- One-line description
- Link to `/docs/projects/{repo}/`

Individual project sub-pages are auto-generated by the sync tool: section index (`_index.md`, frontmatter only), overview page (`overview.md`, README content), and doc sub-pages from `discovery.scan_paths`.

## Specs Browser (`/specs/{repo}/`) — Deferred

For repos that adopt spec-driven development, their `.specify/` artifacts (constitution, spec, plan) would be fetched by the Go sync tool and published as browsable Markdown pages.

This would make specifications visible and discoverable without requiring visitors to navigate raw GitHub files.

**Status**: Deferred to a future feature. The Go sync tool (feature 006) was completed without artifact fetching; this capability was explicitly deferred during spec trimming (2026-03-12).

## Navigation

- **Responsive**: Mobile-first layout, hamburger menu on small screens
- **SEO**: Meta tags, Open Graph tags, sitemap.xml (Doks built-in)
- **Privacy**: Privacy policy page linked in footer (`content/privacy.md`)
- **Brand**: Cyan/teal color palette, DM Sans typography, dark-theme-first aesthetic via custom SCSS overrides
